# 교차검증(Cross-Review) 정책

> 근거: 하네스 엔지니어링 가이드(`output/claude_code_harness_guide.md`) 축 3 — "모델은 자기 자신이 낸 결과물을 잘 평가하는 편향이 있다. 동일 세션 자기평가는 거의 항상 PASS를 낸다."
>
> 개선 Top 3 중 ②번 항목의 본격 운영 정책. Step 4에서 CLAUDE.md에 삽입한 "자기평가 금지" 1줄 규칙을 실무 절차로 확장한다.

---

## 1. 왜 필요한가

- 구현을 수행한 **동일 세션이 자기 산출물을 채점**하면 편향 때문에 거의 PASS로 통과된다.
- 1인 운영 환경에서는 본인이 본인을 검토하는 구조가 되기 쉽고, 이는 사실상 동일 세션 자기평가와 같은 리스크를 가진다.
- 이 정책은 **검토 주체의 분리**와 **검토 형식의 구조화**를 통해 편향을 차단한다.

## 2. 대상 작업

다음 중 **1개 이상** 해당하면 교차검증이 필수다.

- 코드(src / scripts / services) 변경을 수반하는 작업
- 운영 영향이 있는 변경 (배포, 서버 설정, API 키, 스케줄러)
- 전략·파라미터 변경
- CLAUDE.md / settings / infra 변경

### 면제
- 단순 문서 수정, 오타 수정
- 읽기만 수행하는 조사·요약 작업
- 파일 이동·이름 변경 등 기계적 작업

## 3. 검토 주체 옵션 (최소 1개)

구현 세션과 **다른 주체**가 반드시 들어가야 한다.

| 옵션 | 설명 | 언제 쓰나 |
|------|------|----------|
| A. 별도 세션 | 새 Claude Code 세션을 열어 산출물을 검토. 구현 세션의 컨텍스트를 가져오지 않는다 | 소규모 코드/문서 변경 |
| B. 서브에이전트 (`cto` review/gate) | 기존 `cto` 스킬의 review/gate 단계를 호출 | 배포·운영 영향 작업 |
| C. 자동 검증 스크립트 | `scripts/pre_deploy_check.py`, `scripts/qa_validate.py`, `scripts/lint_none_format.py` 등 | 배포 직전, 필수 병행 |
| D. 다른 모델 (선택) | Codex 등 타 모델에 검토 요청 | 품질 중요도가 매우 높은 작업 |

**권장 조합**: 배포 작업은 B + C, 일반 코드 변경은 A + C.

## 4. 검토 결과 기록 형식

**PASS/FAIL 단순 클릭 금지.** 검토가 형식적 승인으로 전락하는 것을 막기 위해 다음 형식으로 기록한다.

```
검증 주체: (A/B/C/D 중 어느 것)
확인 항목: N개 (plans 성공기준 체크박스 기준)
발견 이슈: M개
  - 이슈1: ...
  - 이슈2: ...
판정: PASS / FAIL / 조건부 PASS
조건부 사유(해당 시): ...
```

기록 위치: 해당 작업의 `workspace/plans/YYYYMMDD_작업명.md` 의 "회고" 섹션 또는 "검증 주체" 섹션.

## 5. 1인 운영 환경의 편향 차단 장치

본인이 검토자가 되는 상황은 피할 수 없다. 이때 편향을 최대한 줄이기 위해:

1. **plans 성공기준을 체크리스트로 그대로 사용한다.** 주관 개입을 최소화하기 위해 사전에 정의된 성공기준만 채점한다.
2. **자동 검증 스크립트(C)를 항상 병행한다.** 사람 검토만으로 종결하지 않는다.
3. **검토 세션은 구현 세션과 분리한다.** 동일 대화창에서 연속 검토 금지 — 새 세션에서 plans 파일만 읽고 시작한다.
4. **"발견 이슈 0건 PASS"가 반복되면 편향 신호로 간주한다.** 주간 회고 시 점검.

## 6. 실패 처리

- 검토에서 FAIL이 나오면 수정 후 재검토한다.
- 원인이 반복적·구조적이면 `docs/lessons/`에 기록하고 `pre_deploy_check.py`에 룰을 추가한다 (기존 시행착오 관리 루틴과 동일).
- 동일 원인으로 2회 이상 FAIL 시 해당 영역 작업은 A+B+C 병행 검토로 격상.

## 7. 성공/실패 판정 (정책 도입 효과 측정)

2주 후 다음 지표로 본 정책 자체의 효과를 판정한다.

- **성공**: 대상 작업 100%에 검토 주체 기록 AND 검토에서 발견된 이슈 ≥1건 (정책이 실제로 작동했다는 증거)
- **실패**: 검토 주체 기록률 <70% 또는 모든 검토가 "이슈 0건 PASS"로 통과 (편향 신호)
