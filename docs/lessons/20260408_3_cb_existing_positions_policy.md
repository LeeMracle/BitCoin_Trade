# 서킷브레이커 발동 시 기존 포지션 처리 정책 부재

- **발생일**: 2026-04-08 (발견)
- **심각도**: MEDIUM
- **카테고리**: 전략 / 리스크 관리

## 증상

2026-04-05 21:01 UTC 서킷브레이커 발동 (총자산 300,000 → 212,497, -29.17%). 이후 신규 진입은 차단되었으나 **기존 보유 포지션(RVN/TRX/CHZ) 3건은 그대로 유지**되었음. 3건 모두 각 18,000~20,000원 규모로 사이즈는 작았으나, CB의 목적(손실 확대 방지)에 부합하는지 명확하지 않음.

## 현재 동작

- `services/execution/multi_trader.py`의 CB 체크는 **신규 진입 경로에서만 작동**
- 기존 포지션의 trail_stop 모니터링은 CB와 무관하게 계속 동작
- 결과: CB 발동 후에도 기존 포지션은 정상 로직(trail_stop 히트 시 청산)으로 처리됨

## 쟁점

1. **Option A — 현재 동작 유지 (신규만 차단)**
   - 장점: 이미 진입한 포지션은 리바운드 기회를 가질 수 있음 (실제 4/5~4/8 사이 BCH 회복으로 총자산 285,510까지 복구됨)
   - 단점: "최악의 경우 -20%에서 멈춘다"는 CB 보장이 깨짐. 추가 손실 가능성 상존

2. **Option B — 즉시 전량 청산**
   - 장점: 명확한 최대 손실 상한 보장
   - 단점: 일시적 과매도에서 손절 확정, 이번 케이스에서는 회복분 날릴 뻔함

3. **Option C — 소프트 청산 (가장 큰 MFE/MAE 포지션부터 정리)**
   - 절충안. 구현 복잡도 ↑

## 결정 필요

- [ ] ADR 작성: `docs/decisions/20260408_cb_existing_positions_policy.md`
- [ ] 선택된 옵션 기반으로 `multi_trader.py` CB 핸들러 수정

## 검증 규칙 (Option A 유지 시)

1. CB 발동 시각 이후 신규 진입 0건 확인
2. 기존 포지션의 trail_stop 로직이 계속 동작하는지 확인
3. CB 발동 중 총자산이 초기자본 대비 -25%를 추가로 넘으면 강제 전량 청산 트리거 (2차 안전장치)

## 교훈

서킷브레이커는 "신규 진입 차단"만으로는 불완전하다. 기존 포지션 처리 정책을 명시적으로 정의해야 하며, 명시되지 않은 동작은 리스크 보장이 아니다. 이번 사례는 운 좋게 회복했지만, 하락이 지속되었다면 -29%에서 더 커졌을 것.
