## 1. 배경

알트 펌프 서핑 전략은 P4-04 검토에서 "현물 수수료(0.05%) + 슬리피지로 수익 여지가 좁다"는 이유로 **보류** 판정을 받았다. 2026-04-18 현재 BTC 하락장이 지속되는 상황에서 실행은 의미가 없으며, 이 문서는 **상승장 복귀 후 즉시 착수할 수 있는 설계 청사진**을 미리 정의하는 것을 목적으로 한다.

---

## 2. 펌프 정의 (후보)

알트 펌프를 탐지하는 기준 후보 3가지. 실제 백테스트에서 조합별 성과를 비교한다.

| ID | 정의 | 파라미터 |
|----|------|---------|
| P-A | 1시간 동안 +N% 이상 상승 | N ∈ {5%, 10%, 15%} |
| P-B | 1시간 거래량이 직전 24시간 평균 대비 M배 급증 | M ∈ {3x, 5x, 10x} |
| P-C | F&G 지수가 단기 급등 (ΔFG > 15포인트, 예: 10→25) | ΔFG > 15 within 1h |

> 실제 탐지는 **조합**: P-A AND P-B (상승률 + 거래량 동시 충족)를 1차 필터로, P-C를 강도 보조지표로 활용하는 방식이 과탐지를 줄이는 데 유리할 것으로 가설 설정.

---

## 3. 진입 신호 후보

진입 조건 3가지를 각각 독립적으로 백테스트하여 비교한다.

### Signal-A: 상승률 임계 돌파

```
조건: 현재가 > 직전 1시간 종가 × (1 + threshold)
      threshold ∈ {5%, 10%, 15%}
      + 24h 거래대금 ≥ 5억 KRW (유동성 최소 기준)
진입: 시장가 즉시
```

### Signal-B: 거래량 돌파

```
조건: 해당 1분봉 거래량 > 직전 20분봉 평균 거래량 × vol_mult
      vol_mult ∈ {5, 10, 20}
      + 1시간 수익률 > 0 (방향성 확인)
진입: 시장가 즉시
```

### Signal-C: F&G Momentum

```
조건: F&G 지수가 전 조회 대비 +15 이상 상승 (API 갱신 간격: 약 1시간)
      + BTC/KRW 1h 상승률 > 2%
      + 알트 basket(상위 20종목) 평균 상승률 > 1%
진입: BTC/KRW 시장가 (알트 개별 아닌 BTC 대표 매수)
```

> Signal-C는 F&G API 갱신 지연(최대 1시간) 문제로 실시간성이 낮아 백테스트 결과에 관계없이 실전 적용 난이도가 높다.

---

## 4. 청산 규칙 후보

진입 신호와 독립적으로 조합 가능한 청산 규칙 4가지.

| ID | 규칙 | 비고 |
|----|------|------|
| E-1 | TTL(Time-To-Live): 진입 후 N분 경과 시 무조건 시장가 청산 | N ∈ {15, 30, 60} 분 |
| E-2 | 고점 대비 -X% 트레일링 | X ∈ {1.5%, 2%, 3%} |
| E-3 | 다음 1분봉이 음봉이면 청산 (단봉 반전) | 빠른 청산, 슬리피지 큼 |
| E-4 | 목표 수익 +Y% 도달 시 전량 청산 (고정 TP) | Y ∈ {3%, 5%, 8%} |

> 조합 권장 우선순위: **E-1 + E-2** (TTL 안전망 + 고점 트레일링). E-3 단독은 슬리피지 노출이 크다.

## 5. 위험 요인

### 5.1 시장 구조적 위험

- **펌프 후 덤프 속도**: 알트 펌프는 10분 내 고점 도달 후 -20~-40% 급락이 빈번하다. 진입 타이밍이 고점에 집중될 수 있다.
- **슬리피지**: 업비트 현물에서 단기 급등 종목은 호가창이 얕아 시장가 체결 시 0.3~1%+ 슬리피지 발생 가능.
- **스프레드**: 저유동성 알트의 경우 매수/매도 스프레드 자체가 0.3~0.5%에 달할 수 있다.
- **수수료 부담**: 업비트 taker 0.05% × 2 = 0.10% 왕복. 단기 매매에서 수익률 잠식 비율이 크다.

### 5.2 구현 제약

- **1분봉 데이터 필요**: 현재 시스템은 일봉/4시간봉 데이터만 수집한다. 1분봉 수집 인프라를 별도로 구축해야 한다.
- **실시간 다종목 스캔**: 업비트 전체 KRW 마켓(약 200종목 이상) 동시 모니터링 필요 → rate limit(29 req/sec) 관리 필요.
- **주문 지연**: 신호 감지 → 주문 도달 사이 수백 ms~수 초 지연이 결과에 큰 영향을 줄 수 있다.

### 5.3 비용 가정 (백테스트 적용)

- 수수료: taker 0.05% × 2 = **0.10% 왕복**
- 슬리피지: 시장가 진입/청산 시 **0.1% 추가** (보수적 가정)
- 총 비용 가정: **진입+청산 합산 0.20%**

## 6. 백테스트 설계

### 6.1 데이터 수집 (사전 조건)

현재 일봉/4시간봉 수집 파이프라인을 1분봉으로 확장해야 한다.

```python
# 예시: ccxt로 1분봉 수집 (업비트)
ohlcv = exchange.fetch_ohlcv("BTC/KRW", timeframe="1m", limit=1440)  # 최근 24시간
```

- 저장 위치: `workspace/data/1m/{SYMBOL}/YYYYMM.parquet`
- 수집 대상: 상위 50종목 × 최근 3개월 — 약 1.5억 행
- 수집 스크립트: 신규 작성 필요 (`scripts/collect_1m_data.py`)

### 6.2 백테스트 구현 방식

- `backtest/engine.py` 의 시간 단위를 1분봉으로 교체하거나 별도 `backtest/pump_engine.py` 작성
- 진입 신호별(A/B/C) × 청산 규칙별(E-1~E-4) 조합 실행
- 각 조합의 결과를 `workspace/runs/pump_grid_YYYYMMDD.csv` 에 저장

### 6.3 검증 구간

| 구간 | 기간 | 목적 |
|------|------|------|
| 상승장 IS | 2025-10-01 ~ 2025-12-31 | 전략이 유효한지 기본 확인 |
| 혼조 IS | 2026-01-01 ~ 2026-03-31 | 시장 중립 구간 성과 |
| OOS | 2026-04-01 ~ 착수 시점 | 최종 검증 |

## 10. 주의사항

- **교차검증 필수**: 백테스트 결과는 별도 세션 또는 QA 에이전트로 교차검증. 동일 세션 자기 PASS 금지 ([cross_review_policy.md](../cross_review_policy.md)).
- **업비트 현물 전용**: 선물/공매도 불가. 롱 포지션 중 덤프 발생 시 손절 외 대안 없음.
- **rate limit 준수**: 전체 종목 동시 감시는 29 req/sec 이내로 구성. 웹소켓 활용 권장.
- **레슨 참조**: 봉 마감 기반 전략을 실시간 틱으로 실행 금지 ([lessons/20260329_1](../lessons/20260329_1_tick_vs_bar_entry.md)).
