# Execution Plans

작업 착수 시점의 **실행계획**을 저장하는 디렉토리. 하네스 엔지니어링 가이드(`output/claude_code_harness_guide.md`)의 C7 항목을 충족하기 위해 운영한다.

## 왜 필요한가

- 코드는 git에 남지만 **계획은 휘발**된다.
- 작업이 실패했을 때 "계획이 틀렸는가, 실행이 틀렸는가"를 분리해 회고할 수 있어야 한다.
- 다른 세션·다른 사람이 작업을 이어받을 때의 인수인계 자료가 된다.

## 비자명 작업의 기준

다음 중 **1개 이상** 해당하면 비자명 작업으로 간주하고 plans 파일을 반드시 생성한다.

- 예상 소요 30분 이상
- 코드(src/scripts/services) 변경을 수반
- 외부 시스템 변경(배포, 서버 설정, API 키, DB) 수반
- 전략·파라미터 변경
- CLAUDE.md / settings / infra 변경

단순 문서 수정, 단일 파일 오타 수정, 읽기만 수행하는 조사 작업은 면제한다.

## 파일명 규칙

`YYYYMMDD_작업명.md`

예: `20260409_execution_plan_rollout.md`

같은 날 여러 작업이면 `YYYYMMDD_N_작업명.md` (N은 1부터).

## 작성 절차

1. 작업 착수 직전에 `_TEMPLATE.md`를 복사해 신규 파일 생성
2. 목표·성공기준을 먼저 채운다 (이 부분이 빈칸이면 아직 착수하지 말 것)
3. 작업 수행
4. 작업 종료 시 "회고" 1줄을 채우고 commit

## 회고 시 활용

- 주간 회고(`daily-work weekly`) 시 이번 주 plans/ 목록을 훑어본다.
- 각 파일의 성공기준 체크박스를 기준으로 PASS/FAIL 판정
- FAIL이면 원인 귀속(계획 결함 / 실행 결함)을 구분해 lessons 또는 decisions로 환원
