# Claude Code 프로젝트 운영 가이드 — 하네스 엔지니어링 기반

> 같은 모델·같은 요금제라도 결과물의 격차는 **하네스(Harness)** 설계에서 갈린다.
> 이 문서는 임의의 Claude Code 프로젝트를 점검·개선하기 위한 재사용 가능한 가이드다.

---

## 0. 핵심 개념

### 0.1 하네스란?
**하네스 = 모델을 둘러싼 모든 것.** 마부가 말에 씌우는 고삐·안장·굴레처럼, AI 모델에 **방향(컨텍스트) · 실행력(도구) · 제약(가드레일)**을 부여하는 장비 일체를 말한다. 구성 요소:

- 컨텍스트 (목표, 제약, 컨벤션, 아키텍처)
- 도구 (기본 도구, MCP, 스킬, 커스텀 CLI)
- 실행 루프 (목표 → 작업 → 검증 → 재시도)
- 검증 (테스트, 린트, 평가 기준)
- 사람 (승인, 방향 수정, 최종 판단)

### 0.2 자동화 개선 루프
```
목표 + 검증기준
   ↓
탐색 / 수정 / 생성
   ↓
검증 ──실패──▶ 재탐색
   │
  성공
   ↓
사람 호출 (중간/최종 보고)
```
사람이 매 턴 끼어드는 수동 루프를 끊고, **검증 기반 자가개선 루프**로 전환하는 것이 하네스 엔지니어링의 목표다.

### 0.3 새로운 개발자상
개발자의 주요 업무는 코드 작성이 아니라 **에이전트의 환경·의도·피드백 루프를 설계하는 일**로 이동했다. 코드는 결과물일 뿐이며, 환경 설계의 품질이 곧 산출물의 품질이 된다.

## 1. 3대 축

하네스 품질은 다음 3개 축으로 평가·개선한다.

### 축 1. 컨텍스트 엔지니어링
### 축 2. 도구 & 환경 설계
### 축 3. 평가(Evaluation) 드리븐 개선

각 축은 ① 원칙 → ② 점검 체크리스트 → ③ 보완 가이드 순서로 정리한다.

## 2. 축 1 — 컨텍스트 엔지니어링

### 2.1 원칙

- **컨텍스트는 한정 예산이다.** 1M 토큰을 줘도 약 300K를 넘기면 품질이 급락한다 ("Lost in the middle"). 트랜스포머 구조상 연산량 증가는 곧 오류 확률 증가다.
- **즉시 주입 vs 필요 시 로드**를 구분하라.
  - **즉시 주입**: 작업 목표, 제약사항, 코딩 컨벤션, 최근 아키텍처 변경, 절대 금지사항(보안 경계 등)
  - **필요 시 로드**: 상세 코드 본문, 대용량 로그, 테스트 실행 결과, 에러 트레이스
- **루트 가이드 문서는 "목차"다.** `CLAUDE.md` / `AGENTS.md`는 빡빡한 본문이 아니라 어디에 어떤 문서가 있는지 안내하는 가벼운 인덱스여야 한다. 세부는 분리된 문서로.
- **Execution Plan은 반드시 파일로 저장한다.** 코드는 git에 남지만 계획은 휘발된다. 회고와 책임 추적을 위해 강제 저장.
- **진짜 지켜야 할 규칙은 문서가 아닌 코드/훅으로 강제한다.** 문서로만 두면 안 읽는다.

### 2.2 점검 체크리스트

| # | 항목 | 판정 기준 | 상태 |
|---|------|-----------|------|
| C1 | 루트 가이드 문서 존재 | `CLAUDE.md` 또는 `AGENTS.md`가 레포 루트에 존재 | ☐ |
| C2 | 루트 문서는 "목차형" | 본문 길이가 과도하지 않고, 세부는 별도 문서로 링크 | ☐ |
| C3 | 작업 목표·제약 즉시 주입 | 프로젝트 목표와 금지사항이 루트 문서 상단에 명시 | ☐ |
| C4 | 코딩 컨벤션 명시 | 언어·포맷·네이밍 등 컨벤션 문서 존재 및 링크 | ☐ |
| C5 | 보안 경계 명시 | 접근 금지 폴더·키 권한 제한 등이 명문화 | ☐ |
| C6 | 아키텍처 문서 분리 | 아키텍처/도메인/스펙이 각각 별도 문서로 존재 | ☐ |
| C7 | Execution Plan 강제 저장 | 작업 시작 시 계획을 파일로 남기는 규칙·디렉토리 존재 | ☐ |
| C8 | 시행착오·교훈 누적 | `lessons/` 또는 동등한 폴더에 사후 기록 누적 | ☐ |
| C9 | 컨텍스트 압축 정책 | 자동 압축 활성화 또는 수동 압축 시점 합의 | ☐ |
| C10 | 대용량 로그 격리 | 테스트 결과·로그를 컨텍스트에 직접 붓지 않는 운영 규칙 | ☐ |

### 2.3 보완 가이드

- **C1~C2 미흡**: 루트에 인덱스형 가이드 문서를 새로 작성. 본문은 ① 프로젝트 목표 ② 금지사항 ③ 폴더 구조 ④ 세부 문서 링크의 4블록만 유지.
- **C7 미흡**: `workspace/plans/YYYYMMDD_작업명.md` 같은 위치를 정하고, 루트 가이드에 "작업 시작 전 계획 파일을 생성한다"는 규칙을 명시. 가능하면 훅으로 강제.
- **C8 미흡**: `docs/lessons/` 폴더 신설. 오류 발생 시 (1) 수정 → (2) 기록 → (3) 검증 룰화 → (4) 가이드 갱신 4단계 운영.
- **C10 미흡**: 로그·테스트 산출물은 별도 디렉토리로 격리하고, 에이전트가 필요할 때만 도구로 읽도록 안내.

## 3. 축 2 — 도구 & 환경 설계

### 3.1 원칙

- **승인 기반 루프를 기본값으로.** Bypass / Auto-approve 모드는 초기에는 켜지 말 것. 사용자가 "내 승인 하에 일이 벌어진다"는 감각을 충분히 갖춘 후에만 완화한다.
- **승인 피로 주의.** 같은 승인이 10번 반복되면 사람은 검토 없이 클릭만 한다. → 승인 지점 자체를 줄이도록 도구·범위를 설계하라.
- **기본 도구가 약하면 교체하라.** 예: 기본 WebSearch가 약하면 외부 검색 MCP로 대체.
- **위험 명령은 샌드박스 우선.** 임의 CLI 실행은 위험하다. 안전 경계 안에서 먼저 돌리고, 결과가 안전할 때만 실제 파일·시스템에 반영.
- **금지 행동은 도구로 막아라.** "하지 마라"는 문서보다 권한·훅·사전 검증 스크립트가 강하다.

### 3.2 점검 체크리스트

| # | 항목 | 판정 기준 | 상태 |
|---|------|-----------|------|
| T1 | 승인 모드 정책 | Bypass 모드 사용 여부와 그 이유가 합의·문서화 | ☐ |
| T2 | 위험 명령 가드 | 배포·삭제·외부 호출에 사전 검증 또는 승인 강제 | ☐ |
| T3 | 사전 검증 스크립트 | `pre_deploy_check` 같은 자동 점검 스크립트 존재 | ☐ |
| T4 | 훅(Hooks) 활용 | 반복 강제 사항이 settings 훅으로 자동 실행 | ☐ |
| T5 | 도구 품질 점검 | 기본 도구 중 약한 것을 대체 도구로 교체 (예: 검색) | ☐ |
| T6 | 시크릿 격리 | API 키 등은 `.env`/시크릿 매니저로 분리, 레포 미커밋 | ☐ |
| T7 | 권한 최소화 | 외부 시스템 권한이 작업에 꼭 필요한 범위로 제한 | ☐ |
| T8 | MCP·스킬 인벤토리 | 사용 중인 MCP 서버·스킬 목록과 용도가 문서화 | ☐ |
| T9 | 세션 시작 자동 주입 | 핵심 규칙이 세션 시작 시 자동으로 컨텍스트에 진입 | ☐ |
| T10 | 승인 지점 최소화 | 동일 작업에서 승인이 N회 이상 반복되면 묶거나 자동화 | ☐ |

### 3.3 보완 가이드

- **T1~T2 미흡**: 첫 1~2주는 승인 모드 강제. 이후에도 파일 쓰기·삭제·네트워크·배포 카테고리는 승인 유지.
- **T3~T4 미흡**: 반복 실수가 발생한 영역부터 사전 검증 스크립트를 만들고, 훅으로 자동 실행. 실패 시 작업 차단.
- **T8 미흡**: `tools.md` 또는 동등 문서에 MCP 서버·커스텀 스킬 목록과 호출 조건을 한 페이지로 정리.
- **T9 미흡**: 강제 주입할 규칙을 루트 가이드 또는 훅 기반 메시지로 옮긴다.

## 4. 축 3 — 평가(Evaluation) 드리븐 개선

### 4.1 원칙

- **느낌으로 개선하지 마라.** 산출물과 그 산출 과정 모두를 측정·기록한다.
- **자기 세션 자기평가는 편향이다.** 동일 세션에서 모델에게 자기 작업을 채점시키면 거의 항상 "잘 끝났다"고 답한다.
- **교차검증을 도입하라.** 다른 모델, 다른 세션, 다른 에이전트가 검토하도록 분리. 예: 모델 A로 계획·구현, 모델 B로 리뷰, 다시 A로 수정.
- **하네스 + 모델을 함께 평가하라.** 동일한 모델도 하네스 차이만으로 결과가 극단적으로 갈린다 (하네스 없음: 비용 적고 미작동 / 잘 짠 하네스: 비용 크되 다기능·실제 동작).
- **측정 없으면 평가 없고, 평가 없으면 개선 없다.** 세션 비용·시간·실패 지점을 추적하라.

### 4.2 점검 체크리스트

| # | 항목 | 판정 기준 | 상태 |
|---|------|-----------|------|
| E1 | 평가 기준 정의 | 작업별 성공 기준(Acceptance Criteria)이 작업 시작 전 정의 | ☐ |
| E2 | 산출물 기록 | 결과물(코드/문서/지표)이 git 또는 평가 폴더에 누적 | ☐ |
| E3 | 프로세스 기록 | 작업 계획·중간 판단·실패 사유가 외부 파일에 기록 | ☐ |
| E4 | 교차검증 운영 | 동일 세션 자기평가 금지, 다른 세션/모델/에이전트가 리뷰 | ☐ |
| E5 | 회고 주기 | 일일/주간 단위 회고와 교훈 갱신 루틴 존재 | ☐ |
| E6 | 측정 지표 | 비용/시간/실패율 등 운영 지표를 어떤 형태로든 추적 | ☐ |
| E7 | 실패 → 룰화 경로 | 실패가 검증 룰·훅·문서에 자동으로 환원되는 흐름 존재 | ☐ |
| E8 | 멀티에이전트 분업 | 계획/구현/검토 역할이 분리되어 운영 | ☐ |
| E9 | 평가 산출물 위치 | 평가·실험 결과를 모으는 전용 디렉토리 존재 | ☐ |
| E10 | 환경 변경 추적 | 도구·프롬프트·하네스 변경 이력이 기록·비교 가능 | ☐ |

### 4.3 보완 가이드

- **E1 미흡**: 작업 착수 전 "성공 기준 N개"를 Execution Plan 상단에 적게 한다.
- **E4 미흡**: 리뷰 전용 서브에이전트 또는 별도 모델/세션 운영. "이 세션에서 만든 코드를 같은 세션이 평가하지 않는다"를 규칙화.
- **E7 미흡**: 실패 발생 시 (1) 수정 (2) 교훈 기록 (3) 검증 스크립트에 룰 추가 (4) 가이드 반영의 4단계 루틴을 명문화.
- **E9 미흡**: `evals/` 또는 `experiments/` 디렉토리 신설, 결과를 외부에 누적해 세션을 넘나드는 비교가 가능하게 한다.

## 5. 모델·에이전트 운영 권고

원문은 Claude Code와 Codex의 강점을 다음과 같이 본다 (참고용, 시점에 따라 변할 수 있음).

- **Claude Code 계열**: 코드 작성에 강한 "페어 프로그래머". 명확한 티켓·계획이 주어지면 강력.
- **Codex 계열**: 계획 수립과 컨텍스트 탐색, 검증 루프 오케스트레이션에 강함.
- **권장 패턴**: 한쪽 모델로 계획 → 다른 쪽 모델로 검토·수정 → 다시 첫 모델로 구현. **같은 모델이 만든 것을 같은 모델이 검증하지 않게** 한다.

핵심은 특정 모델 선택이 아니라 **교차검증 구조 자체**다. 동일 모델만 쓰더라도 세션·역할을 분리하면 효과를 얻을 수 있다.

## 6. 팀 운영

- **공통 규칙은 별도 레포로** 분리해 git submodule로 주입한다. 새 프로젝트에도 동일 기준이 자동 적용된다.
- **진짜 강제할 규칙은 문서가 아닌 코드로** 주입한다 (세션 시작 시 강제 로딩, 훅, 사전 검증 스크립트).
- **누구나 이어받을 수 있어야 한다.** Execution Plan과 lessons가 외부 파일로 남아 있어야 인수인계가 가능하다.

## 7. 빠른 자가진단 (요약)

다음 10개 질문에 "예"가 7개 미만이면 하네스 보강이 필요하다.

1. 루트에 인덱스형 가이드 문서가 있는가?
2. 작업 시작 전 Execution Plan을 파일로 남기는가?
3. 보안 경계와 금지 행동이 명문화되어 있는가?
4. 위험 명령에 사전 검증 또는 승인이 강제되는가?
5. 반복 실수 영역에 사전 검증 스크립트가 존재하는가?
6. 시크릿이 레포 외부로 분리되어 있는가?
7. 작업의 성공 기준이 사전에 정의되는가?
8. 자기 세션 자기평가를 피하는 운영 규칙이 있는가?
9. 실패가 교훈·검증 룰로 환원되는 경로가 있는가?
10. 세션 비용/시간/실패 등 어떤 형태로든 측정되고 있는가?

## 8. 적용 절차 (신규/기존 프로젝트 공통)

1. 본 가이드의 3개 체크리스트(C/T/E)를 복사해 현재 프로젝트 상태를 ☐/△/☑로 표시한다.
2. ☐ 또는 △ 항목 중 영향이 큰 3개를 선정한다 (모두 한 번에 고치지 말 것).
3. 각 항목에 대해 보완 가이드를 따라 산출물(파일/스크립트/규칙)을 만든다.
4. 1~2주 운영 후 다시 점검하고, 개선 효과를 짧게 기록한다.
5. 새 실패가 나오면 → 수정 → 교훈 기록 → 검증 룰화 → 가이드 갱신.

*본 가이드는 하네스 엔지니어링 원문(영상 스크립트)을 바탕으로 Claude Code 프로젝트 운영을 위해 재구성되었다. 특정 프로젝트·도메인에 종속되지 않으며, 모든 항목은 객관적 판정과 보완이 가능하도록 설계되었다.*
